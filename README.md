# 🛍️ 3차 미니 프로젝트
 
> *단순 키워드 추출·긍정/부정 집계에 머물던 수작업 분석의 한계를 넘어, 화장품 리뷰에서 속성별 감성 정보를 자동으로 추출하여 제품 기획·마케팅·고객 응대 등 비즈니스 전반의 의사결정에 활용할 수 있는 데이터를 제공한 ABSA Agent 프로젝트*
 
---

## 📌 프로젝트 개요
* **수행 기간**: 2026.05.11 ~ 2026.05.13 
* **팀 구성**: 👥 팀 프로젝트 / 9인
* **사용 데이터**: 화장품 리뷰 샘플 데이터 200건
* **담당 업무**:
    * Supervisor Agent 고도화 — reason_code 기반 재시도 정책 설계 및 repair_directive 생성 (`supervisor_node`)
    * Analyzer Agent 고도화 — 1단계 기본 속성 → 2단계 확장 속성 매핑 프롬프트 개선 (`analyzer_node`)
    * Critic Agent 고도화 — reason_code 4종 분류 체계 및 검증 로직 구현 (`critic_node`)
* **사용 기술**:
    * `GPT-4.1-mini` — 대량 배치 처리·반복 루프 구조에서 비용 효율성을 확보하면서도 JSON 구조화 출력 안정성을 유지하기 위해 선택 
    * `LangGraph` — Analyzer → Critic → Supervisor의 조건부 순환 흐름을 노드 그래프로 명확하게 구조화하고, reason_code 기반 재시도·종료 분기를 State로 제어하기 위해 선택
    * `LangSmith` — Agent 실행 흐름을 Run(Trace) 단위로 추적하고, 실패 지점·재시도 루프·병목 구간을 파악하여 디버깅 효율을 높이기 위해 선택
    * `SQLite` — 별도 DB 서버 없이 분석 결과를 로컬 파일 기반으로 저장·관리하기 위해 선택
    * `Streamlit` — 별도 프론트엔드 개발 없이 분석 결과 집계 및 건별 조회 대시보드를 빠르게 구축하기 위해 선택

* **문제 배경**:
    * 쇼핑라운지 입점 업체의 리뷰와 피드백이 증가하고 있지만, 기존 수작업 분석은 단순 긍정/부정 비율 집계에 불과하여 세부 만족·불만 요인 파악에 한계가 있음
    * 보습, 향 등 화장품 특유의 속성(aspect)에 따라 소비자 감정이 다르게 표현되는데, 이를 속성 단위로 구분하지 못해 제품 개선에 반영하기 어려움
* **비즈니스 요구사항**:
    * 리뷰 데이터를 입력하면 속성별 감성 결과를 자동으로 생성하는 **분석 자동화** 확보
    * 수작업 분류 인력 없이 분석을 자체 처리할 수 있는 **비용 절감**
    * 동일한 기준으로 반복 적용하여 분석 **품질을 균일하게 표준화**
 
* **해결 목표**:
    * 리뷰 데이터를 입력하는 것만으로 분석 → 검증 → 재시도 → DB 저장까지 전 과정을 자동화하는 ABSA Agent 구축
    * Critic Agent가 reason_code 기반으로 오류를 분류하고, repair_directive로 수정 지시를 전달하는 품질 보장 구조 구현
    * 미처리 리뷰를 일괄 처리하는 배치 실행 파이프라인 구축
    * 분석 결과를 속성별로 시각화하는 Streamlit 대시보드 제공

## ✨ 주요 기능
 
### 🔍 속성 기반 감성 분석 (Analyzer)
단순 긍정/부정 분류를 넘어 리뷰에서 속성별 감성을 추출하여 제품 개선에 활용 가능한 세부 인사이트를 생성
- 보습 / 가격 / 향 / 포장 화장품 속성(aspect) 자동 추출 (1단계 기본 속성 → 2단계 확장 속성 순으로 매핑)
- aspect별 감성 라벨 분류 (1=긍정, 0=부정)
- 판단 근거(evidence)를 리뷰 원문에서 직접 추출
- 이전 Critic의 repair_directive를 반영하여 재분석 수행
### ✅ 분석 결과 검증 (Critic)
생성된 분석 결과를 자동으로 검증하고 reason_code를 분류하여 분석 품질을 보장
- `OUTPUT_ERROR` — JSON 형식·스키마 오류, label 범위 오류
- `SCOPE_ERROR` — 허용 범위 밖 속성 사용, 동의어 처리 실패
- `EVIDENCE_ERROR` — 근거 문구가 원문과 불일치, 요약 형태, 근거 없음
- `QUALITY_ERROR` — 환각(Hallucination), 감성 판단 모호, 분석 품질 저하
- 부적합 시 `REPAIR_TEMPLATES` 기반 수정 지시(`repair_directive`) 자동 생성
### 🔀 실행 흐름 제어 (Supervisor)
reason_code 기반 재시도 정책으로 파이프라인 전체 흐름을 제어
- Analyzer / Critic 실행 여부 확인 후 다음 Agent 라우팅
- `RETRYABLE` 코드(`OUTPUT_ERROR` / `SCOPE_ERROR` / `EVIDENCE_ERROR`) 여부 판단
- 최대 재시도 횟수(`max_retries=3`) 초과 시 강제 종료
### 🗄️ 배치 처리 및 DB 저장
미처리 리뷰를 일괄 처리하고 결과를 SQLite DB에 자동 저장하여 대량 리뷰 분석을 자동화
- `get_unprocessed_reviews()`로 `agent_aspect=[]`인 미처리 리뷰 자동 조회
- 리뷰별 Agent 실행 및 오류 발생 시 다음 리뷰로 자동 계속
- `update_result()`로 aspect / label 결과를 SQLite DB에 저장
### 📊 분석 결과 대시보드 (Streamlit)
배치 분석 결과를 속성별로 시각화하고 건별로 조회할 수 있는 웹 대시보드 제공
- 리뷰 직접 입력 후 버튼 클릭 → DB 저장 및 Agent 분석 실행
- 리뷰 분석 결과 건별 조회 (원문 + 속성 + 긍/부정 출력)
- 속성별 집계 결과를 표와 그래프로 확인
- 속성 필터 추가/제거 기능

## 💡 배운 점 및 개선할 점
 
#### ✅ 배운 점
- LangGraph의 StateGraph를 활용해 Analyzer → Critic → Supervisor의 순환 흐름을 노드 단위로 설계하고, reason_code 기반 조건부 분기로 재시도와 종료를 제어하는 구조를 직접 구현해보면서 Agent 파이프라인 설계 감각을 익혔다.
- Critic을 단순 LLM 평가가 아닌 규칙 기반 코드 검증(`OUTPUT_ERROR` 등)으로도 구현할 수 있다는 점을 배웠고, evidence 원문 포함 여부처럼 명확한 조건은 코드로 처리하는 것이 더 안정적임을 체감했다.
- 같은 리뷰라도 LLM이 매번 다른 형식으로 출력할 수 있다는 점을 체감했고, 출력 형식을 JSON으로 고정하고 `OUTPUT_ERROR`로 검증하는 구조를 통해 안정적인 출력을 보장할 수 있음을 배웠다.
- Agent 파이프라인을 구현하기 전에 워크플로우 구조, State 설계, 노드 분리 기준 등을 사전에 정의하는 과정이 생각보다 어려웠지만, 이 설계 단계를 충실히 거치지 않으면 노드 간 데이터 흐름이 꼬이거나 기능 추가 시 구조 전체를 수정해야 하는 문제가 생긴다는 것을 직접 경험하면서 개발 전 설계의 중요성을 체감했다.

#### 🚀 개선할 점
- 현재는 리뷰 분석 자동화에 머물고 있지만, 분석 결과를 기반으로 한 리뷰 기반 상품 추천, 자동 요약, VOC 클러스터링 등 고도화된 서비스로 확장할 수 있는 가능성이 있다.
- 기본 속성(보습·가격·향·포장)만으로 커버되지 않는 리뷰를 처리하기 위해 고도화 과정에서 확장 속성을 추가하여 1단계 기본 속성 → 2단계 확장 속성 순으로 매핑하는 구조를 구현했으나, 현재는 확장 속성 범위도 고정되어 있어 입점 업체의 카테고리에 따라 속성을 동적으로 정의할 수 있는 구조로 발전시킬 필요가 있다.
- 현재 Streamlit에서는 리뷰를 한 건씩 수동으로 입력해야 하는 구조로, 데모용으로는 충분하지만 실제 서비스 관점에서는 CSV 파일을 업로드하면 여러 리뷰를 한 번에 분석할 수 있는 기능을 추가할 필요가 있다.
- 오류 또는 해당 속성 없음으로 미처리된 리뷰는 `agent_aspect=[]` 상태로 DB에 남아 배치를 재실행하면 다시 조회되는 구조이지만, 재실행을 수동으로 해야 한다는 한계가 있다. HITL 노드를 추가하여 사람이 직접 검토하거나, 실패 리뷰만 별도 DB에 저장하여 자동으로 재처리하거나, 매핑 불가 시 기타 속성으로 분류하는 폴백 처리 등을 통해 미처리 리뷰 없이 배치가 완료될 수 있는 구조로 개선할 필요가 있다.
